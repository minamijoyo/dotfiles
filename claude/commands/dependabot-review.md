指定されたブランチ `$1` の依存ライブラリバージョンアップPRについて、影響調査とコード修正を行ってください。

## 処理手順

### 1. PRの情報確認
- `gh pr view $1` でPR情報を取得
- `gh pr diff $1` で変更差分を確認
- アップグレード対象のライブラリと変更前後のバージョンを特定

### 2. ブランチのチェックアウト
```bash
git fetch origin
git checkout $1
```

### 3. コードベースでのライブラリ利用方法の調査
- Grepで該当ライブラリの使用箇所をコードベース全体から検索
- 使用されているAPI、メソッド、引数、パラメータを把握
- 利用パターンを整理（どの機能がどのように使われているか）

### 4. リリースノートの調査
- WebFetchで公式リリースノートを調査
- アップグレード前後の全バージョンの変更点を確認
- **特にステップ3で確認した利用箇所に関連する変更点に注目**：
  - Breaking changes（破壊的変更）
  - Deprecations（非推奨化）
  - API changes（メソッド、引数、戻り値の変更）
  - 依存関係の要件変更

### 5. 影響分析とコード修正
- リリースノートの変更点とコードベースの利用箇所を照合
- 影響のあるコードを特定して修正
- 型チェック・リントを実行して問題を確認

### 6. テスト実行
- テストを実行して全て成功することを確認
- 失敗があれば原因を調査して修正

### 7. コミット
- 修正内容を簡潔に説明するコミットメッセージで変更をコミット
- リリースノートや関連情報への参照を含める

### 8. 調査結果のPRコメント投稿
`gh pr comment $1` で以下を含むMarkdownコメントを投稿：
- **アップグレード内容**: ライブラリ名、変更前後のバージョン
- **コードベースでの利用状況**: 使用箇所と利用パターンの概要
- **主要な変更点**: リリースノートから確認した重要な変更（特に利用箇所に関連するもの）
- **影響と対応**: 修正した箇所（ファイル名:行番号形式）または影響なしの旨
- **検証結果**: 型チェック、リント、テストの結果
- **今後の対応**: 改善提案やissue化すべき項目があれば記載

## 注意点
- リリースノートは公式ドキュメントを優先し、根拠を示す
- コードベース全体への影響を漏れなく調査
- 修正が広範囲に及ぶ場合は別issueとして切り出すことを検討
